# Claim Verification 

## 1. Overview

The final stage of the pipeline formulates claim verification as a **Natural Language Inference (NLI)** task. Given a claim and a set of questions and evidences, the objective is to classify the claim into one of three categories:

* **Supported (True)** — evidence confirms the claim
* **Refuted (False)** — evidence contradicts the claim
* **Conflicting** — evidence contains both supporting and contradicting signals

Formally, given a claim c question set Q =  {q1, q2, ..., qk} and evidence set E = {e1, e2, ..., ek}: 
                        
                        f(c,Q,E) → y 

where y ∈ {True, False, Conflicting}. NOTE: There can be multiple evidences per question.

---

## 2. Input Representation

Each instance is constructed by combining:

1. Claim
2. Decomposed questions
3. Retrieved evidence snippets

### Transformer Input (Baseline Models)

```id="input_fmt"
[Claim]: <claim>
[Questions]: <q1 q2 ...>
[Evidences]: <e1 e2 ...>
```

### Instruction-based Input (Qwen Model)

```id="llm_fmt"
You are a fact-checking expert.

Claim: <claim>

Evidence 1:
Question: <q>
Answer: <doc>

...

Based on the evidence above, classify this claim.
```

### Design Rationale

* Questions provide **reasoning structure**
* Evidence provides **factual grounding**
* Instruction prompts enable **reasoning-style inference** in LLMs

## 4. Baseline Model: RoBERTa-Base

We begin with **roberta-base** from Hugging Face Transformers.

### Characteristics

* 12-layer transformer
* Efficient and lightweight
* Suitable for rapid prototyping

### Training Setup

* Framework: PyTorch
* Optimizer: AdamW
* Learning rate: (2 \times 10^{-5})
* Max length: 512
* Batch size: 8
* Scheduler: linear warmup
* Gradient clipping: enabled

### Experiments

| Model   | Dataset          |
| ------- | ---------------- |
| Model 1 | Our dataset      |
| Model 2 | Merged dataset   |
| Model 3 | QuantEmp dataset |

---

## 5. Baseline Limitation: Context Truncation

### Maximum Length = 512 tokens

The input often includes:

* multiple evidences
* multiple questions

### Impact

* Truncation removes critical information
* Multi-hop reasoning is impaired
* Conflicting evidence handling becomes unreliable

---

## 6. Advanced Model: FinQA-RoBERTa-Large

To improve reasoning capability, we adopt:

* **FinQA-RoBERTa-Large (MNLI variant)**

### Motivation

* Pretrained on financial QA tasks
* Strong numerical reasoning
* Better structured understanding

### Transfer Learning Pipeline
Quantemp already trained a roberta-large-mnli model on finqa dataset. 

roberta-large-mnli → FinQA fine-tuning → Our fine-tuning

---

## 7. Architecture

* Backbone: roberta-large (24 layers, hidden size 1024)
* Pooling: pooler_output
* Classification Head:

Dropout → Linear (1024 → 768) → ReLU → Linear (768 → 3)
---

## 8. Training Strategy

### Key Settings

* Max length: 512
* Batch size: 16
* Epochs: up to 20
* Early stopping: patience = 2

### Special Design Choices

* Scheduler defined but not stepped
* Gradient clipping disabled

### Layer Freezing

* First 5 layers frozen
* Higher layers fine-tuned

---

## 9. Limitation of Transformer-Based Models

Even with improved reasoning:

### Context Length Bottleneck Persists

* Severe truncation at 256 tokens
* Loss of evidence diversity
* Reduced performance in multi-evidence scenarios

---

## 10. Long-Context Model: Fin-o1-8B

To overcome these limitations, we adopt:

* **Fin-o1-8B**, a large-scale reasoning model

### Advantages

* Large context window (up to 2048 tokens in our setup)
* Instruction-following capability
* Strong reasoning over long inputs

---

## 11. Fine-Tuning with LoRA

Due to the large size (8B parameters), full fine-tuning is infeasible. Instead, we use:

* **LoRA (Low-Rank Adaptation)**

### Configuration

* Rank (r): 16
* Alpha: 32
* Dropout: 0.05
* Target modules:

  * attention projections (q, k, v, o)
  * feedforward projections

### Benefits

* Reduces trainable parameters significantly
* Enables efficient GPU training
* Maintains performance

---

## 12. Training Pipeline

### Model Setup

* Base model loaded in **offline mode**
* Precision: bfloat16
* Optimization: AdamW (8-bit)

### Training Strategy

* Batch size: 2
* Gradient accumulation: 8
* Effective batch size: 16
* Epochs: 3
* Warmup ratio: 0.1
* Gradient checkpointing: enabled

## 16. Key Findings

1. Evidence quality and diversity significantly affect performance
2. Merging datasets improves reasoning via complementary signals
3. Transformer models are limited by fixed context size
4. Domain-specific pretraining (FinQA) improves numerical reasoning
5. Long-context models enable full evidence utilization
6. LoRA enables efficient fine-tuning of large models

---

## 17. Final Insight

The experiments demonstrate a clear progression:

* **RoBERTa-base**  → easy to implement
* **FinQA-RoBERTa** → validates pipeline 
* **Fin-o1-8B** → removes context bottleneck & improves reasoning
